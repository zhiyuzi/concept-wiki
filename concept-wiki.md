# concept-wiki

一种用 LLM 构建和维护个人知识库的模式。

这是一个理念文件，设计用于直接粘贴给你的 LLM Agent（例如 Claude Code、Cursor、OpenCode 等）。它的目标是传达核心思想，具体实现由你和 Agent 协作完成。

## 核心思想

大多数人使用 LLM 处理文档的方式是 RAG：上传文件，检索相关片段，生成回答。这种方式有效，但 LLM 每次都在从零重新发现知识，没有积累。

这里的思路不同。LLM **持续构建并维护一个 wiki**——一套结构化、互相链接的 markdown 文件，位于你和原始资料之间。当你添加新资料时，LLM 不只是索引它，而是读取它、提取关键信息，并将其整合进现有 wiki——更新相关页面，标记新旧内容的矛盾，强化或挑战已有的理解。知识被编译一次，然后持续保持更新，而不是每次查询都重新推导。

这是关键区别：**wiki 是一个持久的、复利增长的产物。** 交叉引用已经在那里。矛盾已经被标记。综合已经反映了你读过的一切。每添加一个资料，每提一个问题，wiki 就变得更丰富。

你几乎不需要亲自写 wiki——LLM 负责写和维护所有内容。你负责选材、探索和提问。LLM 做所有繁琐的工作：摘要、交叉引用、归档和维护。

## 本体

concept-wiki 的底层本体由两条平行的层级构成，通过 Concept 节点相交。

### 知识的结构（SoDK）——陈述性知识

从低到高，抽象层级依次提升，可迁移性依次增强：

- **Fact**：具体的人、地点、情境或事物的特定实例。不可迁移，锁定在特定时间、地点或情境中。
- **Topic**：围绕特定人、地点、情境或事物的一组相关 Fact 的框架。不可迁移。
- **Concept**：从 Topic 中抽象出的心理构造。满足：跨时间成立、一到两个词或短语、普遍性、抽象性。可迁移。Concept 是 SoDK 与 SoPK 的共享桥节点。
- **Generalization**：两个或多个 Concept 构成的关系陈述，跨时间、跨文化、跨情境迁移。通常需要限定词（often、can、may）。
- **Principle**：被视为某学科基础"真理"的 Generalization。不使用限定词。
- **Theory**：用于解释某一现象或实践的一组概念性假设，以最佳证据而非绝对事实为支撑。

### 过程的结构（SoPK）——程序性知识

从低到高，复杂度依次提升：

- **Skill**：嵌入在 Strategy 中的最小操作单元。Skill 支撑 Strategy 运作。Skill 的内容天然适合结构化表达——可以是自然语言描述，也可以是带参数的可执行定义（如函数签名、preconditions、postconditions、failure modes）。具体格式取决于你的使用场景，由 Schema 层约定。
- **Strategy**：系统性计划。包含多个 Skill，需要对这些 Skill 有控制力、流畅性和灵活性。
- **Process**：产生结果的持续行动。Process 是连续的，只有在干预发生时才停止。
- **Concept**：与 SoDK 共享的桥节点。SoPK 的 Concept 既来自内容主题，也来自 Process、Strategy、Skill 本身。
- **Generalization / Principle / Theory**：与 SoDK 定义相同，但证据来源包含程序性知识。

### 层级规则

两套结构共享同一套层级规则：

- **严格相邻**：节点只能与相邻层级的节点建立上下关系，不支持跨层跳跃。
- **方向向上**：关系方向是从下层指向上层——多个下层节点支撑一个或多个上层节点。
- **可迁移性是分界线**：Fact 和 Topic（SoDK）、Skill、Strategy、Process（SoPK）不可迁移，锁定在具体情境中，是临时的上下文支撑；Concept 及以上层级可迁移，跨时间、跨领域成立，是长期稳定资产。这条线对 wiki 的维护有实质影响：lint 对两类节点的检查逻辑不同——下层节点需要检查是否仍然有效，上层节点需要检查是否有足够的下层支撑。

### Concept 是枢纽

Concept 是整个本体的核心节点，也是 SoDK 与 SoPK 的唯一交汇点。多个 Topic 向上支撑 Concept，多个 Skill/Strategy/Process 也向上支撑 Concept；多个 Concept 再向上共同构成 Generalization。Concept 是 wiki 中最值得精心维护的节点类型。

## 架构

三个层次：

**原始资料层（Raw Sources）**——你策划的原始文档集合。文章、论文、图片、数据文件。这些是不可变的——LLM 只读不改。这是你的事实来源。

**Wiki 层**——LLM 生成和维护的 markdown 文件目录。每个节点（Fact、Topic、Skill、Strategy、Process、Concept、Generalization、Principle、Theory）都是一个独立的 markdown 文件，通过 `[[]]` 双向链接按层级关系互相连接。LLM 完全拥有这一层：创建页面、更新页面、维护链接、保持一致性。你来读；LLM 来写。

每个节点文件都应包含 YAML frontmatter。`type` 字段是必要的——它是本体在文件里的唯一机器可读编码，是 lint 做语义检查和 Dataview 做类型查询的基础。其他字段（如 `sources`、`layer`、`created` 等）根据你的需求和 Schema 约定自行决定，保持灵活。

**Schema 层**——一个配置文件（例如 CLAUDE.md、AGENTS.md），告诉 LLM wiki 的结构、命名约定和工作流。这是让 LLM 成为一个有纪律的 wiki 维护者而不是普通聊天机器人的关键。你和 LLM 随时间共同演化这个文件。

## 操作原语

**Ingest（摄入）**。你将新资料放入原始资料集合，告诉 LLM 处理它。示例流程：LLM 读取资料，与你讨论关键要点，从中提取 Fact 和 Topic，识别可抽象的 Concept，判断是否能形成 Generalization 或 Principle，同时识别其中的 Process、Strategy、Skill，将 SoDK 与 SoPK 通过共享 Concept 连接，更新 index，追加 log。一个资料可能触达 10–15 个 wiki 页面。

**Query（查询）**。你向 wiki 提问。LLM 搜索相关页面，读取它们，综合出带引用的回答。回答可以采用不同形式——markdown 页面、对比表格、幻灯片（Marp）、图表。关键洞察：**有价值的回答可以作为新页面回存到 wiki 中。** 你发现的比较、分析、连接——这些不应该消失在聊天记录里。这样你的探索就像摄入的资料一样在知识库中复利积累。

**Lint（健康检查）**。定期让 LLM 检查 wiki 的健康状态。检查项包括：没有 Fact 支撑的 Generalization、没有归属 Strategy 的 Skill、没有归属 Process 的 Strategy、只挂在 SoDK 侧没有对应 SoPK 的 Concept（反之亦然）、孤儿页面、缺失的链接、被新资料推翻的过时内容、应有但尚未建立的节点。Lint 的本质是维护知识图的语义一致性、覆盖率和连通性。

## 索引与日志

两个特殊文件帮助 LLM（和你）在 wiki 增长时导航。

**index.md** 面向内容。它是 wiki 中所有内容的目录——每个页面列出链接、一行摘要和可选元数据（节点类型、日期、来源数量）。按节点类型分类组织。LLM 在每次 ingest 时更新它。查询时，LLM 先读 index，再钻取相关页面。

**log.md** 面向时间。它是一个只追加的事件记录——记录 ingest、query、lint 发生了什么、发生在什么时候。建议每条记录以统一前缀开头（例如 `## [2026-04-20] ingest | 资料标题`），这样 log 可以用简单工具解析。

## 可选工具

在某个规模之后，你可能需要构建小工具帮助 LLM 更高效地操作 wiki。最明显的是搜索引擎——在小规模下 index 文件已经够用，但随着 wiki 增长，你需要真正的搜索。[qmd](https://github.com/tobi/qmd) 是一个好选择：本地 markdown 文件的混合 BM25/向量搜索引擎，带 CLI 和 MCP server。你也可以自己构建更简单的东西——LLM 可以帮你快速写一个朴素搜索脚本。

## 提示与技巧

- **Obsidian** 是这套系统的理想前端。`[[]]` 双向链接天然对应 wiki 节点之间的层级关系；图谱视图让你看到哪些 Concept 是枢纽，哪些节点是孤儿；Dataview 插件可以对 frontmatter 字段做动态查询。
- **Obsidian Web Clipper** 是将网页文章快速转为 markdown 的浏览器扩展，适合快速将资料放入原始资料层。
- **图片本地化**。在 Obsidian 设置中将附件文件夹固定到某个目录（例如 `raw/assets/`），然后绑定快捷键下载当前文件的所有附件。LLM 无法在一次读取中同时处理 markdown 文本和内联图片——变通方法是先读文本，再单独查看图片。
- **wiki 就是一个 git repo**。你免费获得版本历史、分支和协作能力。

## 为什么这有效

维护知识库的繁琐部分不是阅读或思考——而是记账工作：更新交叉引用、保持摘要最新、标记新旧内容的矛盾、在数十个页面间保持一致性。人类放弃 wiki 是因为维护负担增长速度快于价值。LLM 不会感到无聊，不会忘记更新交叉引用，可以在一次操作中触达 15 个文件。wiki 保持维护，因为维护成本接近于零。

人类的工作是策划资料、引导分析、提出好问题，以及思考这一切意味着什么。LLM 的工作是其他一切。

## 注意

这个文件是刻意抽象的。它描述的是模式，不是具体实现。确切的目录结构、schema 约定、页面格式、工具链——这一切都取决于你的领域、你的偏好和你选择的 LLM。上面提到的所有内容都是可选和模块化的——取你需要的，忽略其余的。正确的使用方式是将这个文件分享给你的 LLM Agent，然后共同实例化一个适合你需求的版本。这个文件唯一的工作是传达模式。你的 LLM 可以搞定其余的。
